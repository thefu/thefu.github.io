理解 Tokio：Rust 异步编程入门指南（摘要与实用示例）

Tokio 是 Rust 生态中最流行的异步运行时（async runtime）之一，为基于 async/.await 的并发程序提供执行器、IO 驱动、定时器、任务调度和实用工具。本文总结了 Tokio 的核心概念、常用 API、实战示例及最佳实践。

核心概念

  • Runtime：Tokio 的执行环境，包含一个或多个线程（multi-threaded runtime）或单线程（current-thread runtime）。负责调度 Future 的执行。
  • Reactor / I/O 驱动：基于操作系统的事件通知（如 epoll/kqueue），非阻塞地处理套接字/文件等 I/O 事件。
  • Executor：将就绪的任务排入执行队列并在 runtime 线程上运行。
  • Task：由 Future 封装、由 runtime 调度的执行单元（类似轻量线程）。
  • Waker：当 Future 可继续执行时唤醒对应 task。
  • spawn vs spawn_blocking：非阻塞任务（async）用 spawn，阻塞/CPU 密集任务用 spawn_blocking。

错误处理与取消

  • JoinHandle::abort() 可取消任务，但需注意资源清理。
  • 任务取消会在 .await 点抛出 Cancelled，注意捕获与清理（drop 语义）。

与阻塞/CPU 密集型代码的整合

  • 对于 CPU 密集任务，使用 spawn_blocking 或移动到专门的线程池。
  • 尽量将同步 I/O 放到独立线程或使用 async-compatible 库。

网络性能与 Tunables

  • 使用多线程 runtime 提升并发吞吐。
  • 减少频繁分配、避免过多 await 点导致上下文切换。
  • 通过 tokio::sync::Semaphore 控制并发连接数或并发任务数。

与第三方生态

  • 与 hyper（HTTP）、reqwest（HTTP 客户端）、tokio-postgres、sqlx 等集成良好。
  • tracing 与 tokio-console 可帮助调试运行时性能和任务行为。

常见陷阱

  • 在 async 中使用同步锁（如 std::sync::Mutex）会阻塞线程池，导致死锁或性能下降。
  • 忽略 spawn_blocking 导致 runtime 线程被阻塞。
  • 在多线程 runtime 中共享非线程安全数据（未加 Arc/锁）会出错。
  • 忽视任务数量与系统资源（文件描述符、连接数）限制。

实战建议

  • 从 tokio::main 快速上手，逐步使用 spawn 将并发工作分离。
  • 把短小的异步任务放到 async executor，长时间或阻塞工作放到 spawn_blocking。
  • 使用 tokio::sync 系列原语做异步内部同步。
  • 用 tokio-console 或 tracing 分析热点与阻塞点。
  • 书写可取消、健壮的任务并注意资源清理（Drop 与 finally 模式）。
