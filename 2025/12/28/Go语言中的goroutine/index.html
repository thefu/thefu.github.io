Go 语言中的 goroutine

goroutine 是 Go 语言并发编程中最核心的概念。它可以理解为由 Go 运行时管理的轻量级执行单元，写法上像启动一个函数，运行时则由调度器负责把大量 goroutine 分配到少量操作系统线程上执行。

相比直接使用线程，goroutine 的创建成本更低，栈空间可以按需增长，适合用来处理网络请求、后台任务、并发 I/O、流水线计算等场景。

goroutine 与线程的区别

goroutine 和操作系统线程都可以并发执行，但管理方式不同。

  • 线程由操作系统调度，创建和上下文切换成本相对更高
  • goroutine 由 Go runtime 调度，初始栈更小，可以动态增长
  • Go 调度器会把多个 goroutine 映射到多个系统线程上运行
  • 开发者通常不需要直接管理线程生命周期

Go 调度模型常被概括为 GMP：

  • G：goroutine，表示一个待执行任务
  • M：machine，表示操作系统线程
  • P：processor，表示执行 goroutine 所需的调度资源

理解 GMP 不要求记住所有底层细节，但要知道：goroutine 不是无限资源。创建太多 goroutine 同样会带来内存、调度和 I/O 压力。

等待 goroutine 结束

实际开发中不能依赖 time.Sleep 等待 goroutine。更常见的方式是使用 sync.WaitGroup。

WaitGroup 的使用规则：

  • 启动 goroutine 前调用 Add
  • goroutine 结束时调用 Done
  • 主流程调用 Wait 阻塞等待

通常把 Done 放在 defer 中，避免中途返回时忘记通知。

goroutine 泄漏

goroutine 泄漏是 Go 并发程序中很常见的问题。所谓泄漏，就是 goroutine 启动后一直无法退出，占用内存和调度资源。

常见原因包括：

  • 从 channel 接收数据，但没有人发送
  • 向 channel 发送数据，但没有人接收
  • 后台循环没有退出条件
  • 请求取消后，子 goroutine 没有感知取消信号

小结

goroutine 让 Go 的并发编程非常简洁，但简洁不等于没有成本。使用 goroutine 时要重点关注三个问题：

  • 谁负责等待它结束
  • 它在什么条件下退出
  • 并发数量是否可控

写 Go 并发程序时，启动 goroutine 很容易，真正重要的是设计好生命周期。只要 goroutine 的启动、通信、取消和回收都清楚，程序就会更稳定，也更容易维护。
