Go 语言中的 context 包

context 是 Go 标准库中用于传递取消信号、超时时间、截止时间和请求级数据的包。在服务端开发中，context.Context 几乎无处不在，例如 HTTP 请求处理、RPC 调用、数据库查询、后台 goroutine 控制等。

如果说 goroutine 负责并发执行，channel 负责并发通信，那么 context 很多时候负责控制并发任务的生命周期。

context 解决什么问题

在 Go 程序中，一个请求可能会启动多个 goroutine，并调用多个下游服务。

例如：

  • HTTP 请求进入服务
  • 查询数据库
  • 调用缓存
  • 请求第三方接口
  • 启动 goroutine 并发处理多个子任务

如果客户端断开连接，或者请求已经超时，后面的数据库查询、网络请求和子 goroutine 就应该尽快停止。否则这些任务继续运行，会浪费资源，甚至造成 goroutine 泄漏。

context 主要解决这些问题：

  • 通知子任务取消
  • 设置超时时间
  • 设置截止时间
  • 在调用链上传递请求级数据

Background 与 TODO

创建 context 的根节点通常使用 context.Background()。

Background 一般用于：

  • main 函数
  • 初始化逻辑
  • 测试代码
  • 顶层请求入口

context.TODO() 表示暂时还不确定应该使用哪个 context。

如果代码还没想清楚 context 来源，可以临时使用 TODO，但正式业务逻辑中应尽量传递真实的 context。

WithDeadline

context.WithDeadline 用于设置具体截止时间。

WithTimeout 本质上可以理解为基于当前时间计算出一个 deadline。两者区别在于：

  • WithTimeout 使用一段持续时间
  • WithDeadline 使用一个明确时间点

业务中如果已经有明确的截止时间，例如请求头里传递了 deadline，就可以使用 WithDeadline。

在 HTTP 服务中的使用

Go 的 net/http 请求对象自带 context。

当客户端断开连接、请求被取消，或者服务端超时控制触发时，r.Context() 会收到取消信号。下游函数应该继续接收这个 context，而不是重新创建一个无关的 context。

这样请求取消信号可以沿着调用链继续传递。

常见使用原则

context 作为第一个参数

Go 代码中通常把 context.Context 放在函数的第一个参数。

这样调用方一眼就能看出该函数支持取消和超时控制。

不要把 context 存进结构体

一般不建议把 context 放在结构体字段中。

更推荐在每次调用方法时显式传入：

context 表示一次调用链的生命周期，而结构体通常比一次请求活得更久。把 context 存进结构体容易造成生命周期混乱。

不要传 nil context

如果暂时没有合适的 context，可以使用 context.TODO()，不要传 nil。

传 nil 会导致下游代码调用 ctx.Done()、ctx.Value() 时 panic。

cancel 要及时调用

使用下面这些函数时，记得调用 cancel：

  • context.WithCancel
  • context.WithTimeout
  • context.WithDeadline

即使任务提前完成，也应该调用 cancel，让 runtime 及时释放定时器等资源。

小结

context 是 Go 并发和服务端编程中非常重要的工具。它不负责执行任务，也不负责传递大量业务数据，而是负责表达一次调用的生命周期。

使用 context 时可以记住几条原则：

  • context 通常作为函数第一个参数
  • 从上游传递 context，不要随意重新创建
  • 使用 Done() 监听取消信号
  • 使用 WithTimeout 或 WithDeadline 控制超时
  • 使用 WithValue 只传递请求级元数据
  • 创建可取消 context 后要调用 cancel

goroutine、channel 和 context 经常一起出现。goroutine 执行任务，channel 传递结果，context 控制任务何时停止。把这三者配合好，是写出稳定 Go 并发程序的关键。
