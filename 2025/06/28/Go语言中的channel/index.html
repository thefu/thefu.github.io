Go 语言中的 channel

channel 是 Go 语言中用于 goroutine 之间通信的核心机制。Go 并发编程有一句很经典的话：不要通过共享内存来通信，而要通过通信来共享内存。channel 正是这句话在语言层面的体现。

通过 channel，不同 goroutine 可以安全地发送和接收数据，避免直接共享变量带来的数据竞争问题。

channel 是什么

channel 可以理解为 goroutine 之间传递数据的管道。一个 goroutine 可以向 channel 发送数据，另一个 goroutine 可以从 channel 接收数据。

无缓冲 channel

无缓冲 channel 强调同步。发送方和接收方必须同时就绪，数据才能交接。

对于无缓冲 channel 来说，ch <- 1 不只是发送数据，也表示发送方等待接收方接走数据。因此它经常用于 goroutine 之间的同步。

有缓冲 channel

有缓冲 channel 可以存放一定数量的数据。

当缓冲区未满时，发送不会阻塞；当缓冲区为空时，接收会阻塞。

  • 不要在接收方关闭 channel
  • 不要重复关闭 channel
  • 向已经关闭的 channel 发送数据会 panic
  • 从已经关闭的 channel 接收数据不会 panic，会得到零值和 ok == false

当需要持续读取 channel 中的数据，直到 channel 关闭时，可以使用 for range。

for range 会一直读取 channel，直到 channel 被关闭并且数据被读完。

select 多路复用

select 可以同时等待多个 channel 操作。哪个 case 先准备好，就执行哪个 case。

select 常见用途：

  • 同时监听多个 channel
  • 实现超时控制
  • 响应取消信号
  • 避免 goroutine 永久阻塞

单向 channel

函数参数中可以限制 channel 的方向，提高代码可读性和安全性。

  • chan<- int：只能发送
  • <-chan int：只能接收
  • chan int：既能发送也能接收

单向 channel 常用于明确函数职责。例如生产者只负责写入，消费者只负责读取。

nil channel

未初始化的 channel 是 nil。对 nil channel 发送或接收都会永久阻塞。

实际开发中，nil channel 有时会被用在 select 中动态启停某个 case，但普通业务代码里要谨慎使用。

channel 不是队列的万能替代

channel 可以用作任务队列，但不适合滥用。对于复杂队列能力，例如优先级、持久化、重试、延迟消息等，应使用更合适的数据结构或消息队列系统。

小结

channel 是 Go 并发模型的关键组件。它让 goroutine 之间可以通过消息传递协作，而不是直接争抢共享变量。

使用 channel 时要重点关注：

  • 谁发送数据
  • 谁接收数据
  • 谁负责关闭 channel
  • 是否可能阻塞
  • 是否需要超时或取消机制

channel 写法简单，但设计不好很容易出现死锁和 goroutine 泄漏。清楚地定义通信方向和生命周期，是写好 Go 并发程序的基础。
